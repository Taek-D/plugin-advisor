# Architecture

**Analysis Date:** 2026-03-11

## Pattern Overview

**Overall:** Layered client-server architecture with keyword-matching recommendation engine and optional AI analysis.

**Key Characteristics:**
- Separation of concerns: UI layer, analysis layer, data layer
- Plugin recommendation via keyword matching or Claude AI analysis
- Real-time conflict detection and setup readiness checks
- Support for multiple input modes (text, file, GitHub README)
- Admin panel for plugin and suggestion management

## Layers

**Presentation Layer (Frontend):**
- Purpose: React client components for user interaction, analysis flow, and results display
- Location: `components/`, `app/page.tsx`, `app/advisor/page.tsx`
- Contains: Input interfaces, plugin cards, modals, admin forms, preset packs
- Depends on: Hooks (`useAnalysis`), library utilities, i18n
- Used by: Browser clients

**Analysis Layer:**
- Purpose: Core recommendation logic and analysis processing
- Location: `lib/recommend.ts`, `hooks/useAnalysis.ts`
- Contains: Keyword matching algorithm, scoring functions, preset pack detection, confidence calculation
- Depends on: Plugin database, conflict rules, setup helpers
- Used by: Client components, API endpoints

**API Layer:**
- Purpose: Server-side operations (external integrations, rate limiting, AI queries)
- Location: `app/api/`
- Contains: Routes for AI analysis, GitHub fetch, admin operations, lead capture
- Depends on: Anthropic SDK, Next.js server context, authentication
- Used by: Frontend via HTTP requests

**Data Layer:**
- Purpose: Plugin database, types, and static configuration
- Location: `lib/plugins.ts`, `lib/types.ts`, `lib/presets.ts`, `lib/conflicts.ts`
- Contains: 50+ plugin definitions, type definitions, conflict rules, preset packs
- Depends on: None (pure data)
- Used by: All layers

**Infrastructure Layer:**
- Purpose: Utilities, authentication, session management, monitoring
- Location: `lib/` (admin-session, rate-limit, analytics, i18n)
- Contains: Session helpers, rate limiting, tracking, internationalization
- Depends on: External services (Supabase for admin)
- Used by: API routes and client components

## Data Flow

**Keyword-Based Analysis Flow:**

1. User enters text/file/GitHub URL via InputPanel (`components/InputPanel.tsx`)
2. `PluginAdvisorApp` calls `useAnalysis().handleAnalyze()`
3. `recommend(text)` in `lib/recommend.ts` executes:
   - Extracts keywords from input text
   - Detects preset pack intent (beginner-essential, webapp-starter, backend-start, data-research)
   - Scores all plugins against keywords and preset
   - Applies trust scoring based on verification status, install mode, difficulty
   - Returns top 3 recommendations with matched keywords
4. Client displays recommendations in `PluginCard` components
5. User selects plugins (toggles), system checks conflicts via `getConflicts()`
6. `InstallScript` generates install commands for selected plugins

**AI Analysis Flow:**

1. User selects AI mode and submits text
2. Frontend calls `POST /api/analyze` with text and optional candidate plugin IDs
3. Server-side (`app/api/analyze/route.ts`):
   - Rate limits request (5 per minute)
   - Builds system prompt with available plugins
   - Calls Claude Sonnet 4 API
   - Validates response JSON
   - Filters recommendations against allowed plugins
   - Returns AnalysisResult
4. Client processes response identically to keyword flow

**Admin Flow:**

1. User navigates to `/admin/login`
2. Admin auth check via `lib/admin-session-core.ts` (password comparison with trim)
3. `/admin/plugins` → CRUD operations on plugin database (Supabase)
4. `/admin/suggestions` → Review user plugin suggestions
5. Changes persisted to Supabase tables

**State Management:**

- **Client State:** `useAnalysis` hook manages analysis step, results, selected plugins, detail panel
- **Local Storage:** History entries, favorites, session data
- **Supabase:** Admin session validation, plugin database, plugin suggestions
- **Request-scoped:** Versions fetched from GitHub releases API

## Key Abstractions

**Plugin:**
- Purpose: Represents a Claude Code plugin with metadata, keywords, installation info, and operational details
- Examples: `omc`, `superpowers`, `bkit`, `context7` defined in `lib/plugins.ts`
- Pattern: Static plugin database with operational field overrides per plugin

**Recommendation:**
- Purpose: Links a plugin to input context with priority and matched keywords
- Examples: `{ pluginId: "omc", priority: 1, reason: "...", matchedKeywords: ["multi-agent"] }`
- Pattern: Generated by either keyword matching or AI analysis

**AnalysisResult:**
- Purpose: Complete analysis output including recommendations, warnings, setup checks
- Examples: Contains summary, recommendations, preflightChecks, setupWarnings, notRecommended
- Pattern: Returned by both `recommend()` and `/api/analyze`

**PresetPack:**
- Purpose: Predefined plugin combinations targeting specific user types
- Examples: `beginner-essential`, `webapp-starter`, `backend-start`, `data-research`
- Pattern: Configured in `lib/presets.ts` with exclusion rules and intent signals

**ConflictWarning:**
- Purpose: Alerts when incompatible plugins are selected together
- Examples: OMC ↔ Superpowers, Playwright ↔ Puppeteer, Brave Search ↔ Tavily
- Pattern: Pair-based rules defined in `lib/conflicts.ts`

## Entry Points

**Public Routes:**
- Location: `app/page.tsx` (landing page)
- Triggers: Direct visit or internal links
- Responsibilities: Introduces product, explains philosophy, links to advisor and guides

**Advisor Route:**
- Location: `app/advisor/page.tsx`
- Triggers: User clicks "시작 진단"
- Responsibilities: Main analysis interface, loads PluginAdvisorApp

**Admin Routes:**
- Location: `app/admin/login/page.tsx`, `app/admin/plugins/page.tsx`, `app/admin/suggestions/page.tsx`
- Triggers: Admin navigation
- Responsibilities: Authentication, plugin management CRUD, suggestion review

**API Routes:**
- Location: `app/api/analyze/route.ts`, `app/api/github/route.ts`, `app/api/admin/*`
- Triggers: Frontend HTTP POST requests
- Responsibilities: External API calls, server-side logic, database operations

## Error Handling

**Strategy:** Graceful fallback with user-friendly error messages in Korean/English.

**Patterns:**
- **API errors:** Return NextResponse.json with error message and HTTP status code
- **Client errors:** Catch in try-catch, display error toast or inline message
- **Rate limiting:** 429 response with retry-after guidance
- **Validation:** Check input type/length before processing; validate plugin IDs against whitelist
- **GitHub fetch:** Fallback to error message suggesting manual paste

## Cross-Cutting Concerns

**Logging:** `lib/analytics.ts` tracks events (landing_view, preset_select, plugin_detail_view, plugin_suggestion_submit) via Supabase

**Validation:**
- Plugin ID whitelist in recommendation validation
- Text length limit (10,000 chars) for AI analysis
- GitHub URL regex matching

**Authentication:**
- Admin routes protected by password check (`lib/admin-session-core.ts`)
- Session stored in localStorage with 24-hour expiry
- Trim handling for password comparison to prevent whitespace bypass

**Internationalization:**
- Context provider `I18nProvider` in layout
- Locale toggle in Nav component
- Strings in `lib/i18n/ko.ts` and `lib/i18n/en.ts`
- Plugin database includes both Korean and English descriptions
